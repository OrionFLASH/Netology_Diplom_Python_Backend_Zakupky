# Архитектура решения

## 1. Назначение системы

Backend обслуживает сценарий заказа товаров из каталога, сформированного из прайсов нескольких магазинов. Покупатель работает через REST API: регистрация, каталог, корзина (заказ в статусе «корзина»), адреса доставки, подтверждение заказа. Магазин обновляет прайс (YAML), может отключать приём заказов и просматривать заказы, где есть его товары.

## 2. Структура каталогов и ответственность модулей

### 2.1. Корень репозитория

- **`manage.py`** — единственная точка входа CLI; перед запуском Django в `sys.path` добавляется каталог `src`, чтобы пакет `catalog` импортировался как приложение верхнего уровня (рядом с `zakupki`).
- **`zakupki/`** — конфигурационный пакет Django-проекта: `settings.py`, корневой `urls.py`, `wsgi.py`, `asgi.py`, интеграция **Celery** (`celery.py`, импорт в `__init__.py`), вспомогательный **`logging_utils.py`** (форматтер для DEBUG-логов).
- **`src/catalog/`** — основное приложение предметной области.
- **`src/catalog/services/`** — чистая бизнес-логика без привязки к HTTP: импорт/экспорт YAML, текст писем и вызовы `django.core.mail`.
- **`src/catalog/views_*.py`** — классы API (DRF `APIView` / `ListAPIView`), сгруппированные по темам: аутентификация, каталог, корзина, контакты, заказы, партнёр.
- **`src/catalog/serializers.py`** — представление моделей в JSON для ответов и частично для валидации входящих данных.
- **`src/catalog/signals.py`** — реакции на события: регистрация, сброс пароля, смена статуса заказа (в т.ч. переход из корзины в «новый»).
- **`src/catalog/tasks.py`** — задачи Celery для фоновой отправки писем и импорта.
- **`src/catalog/privacy.py`** — функции маскирования данных перед записью в лог.
- **`src/Tests/`** — автотесты на `pytest-django`.

Такое разбиение позволяет проверяющему быстро найти: модели — в `models.py`, HTTP — в `views_*.py`, побочные эффекты (почта, импорт файлов) — в `services/` и `signals.py` / `tasks.py`.

## 3. Модель данных (связи)

### 3.1. Пользователь и магазин

- **`User`** (кастомная модель, `AUTH_USER_MODEL`) — вход по **email**; поле **`type`**: `buyer` или `shop`; новые учётные записи покупателя создаются с **`is_active=False`** до подтверждения email.
- **`Shop`** — привязан к пользователю типа `shop` связью один-к-одному (`related_name="shop"`); поле **`state`** — принимает ли магазин заказы.

### 3.2. Каталог

- **`Category`** — название; связь **M2M** с `Shop` (категория может относиться к нескольким магазинам, как в учебном сценарии импорта).
- **`Product`** — номенклатура (название, категория, опционально **описание**).
- **`ProductInfo`** — конкретное предложение: товар + магазин + **внешний id** из прайса, цена, РРЦ, остаток, модель; ограничение уникальности `(product, shop, external_id)`.
- **`Parameter`** / **`ProductParameter`** — именованные характеристики (произвольный набор имён и значений на каждое `ProductInfo`).

### 3.3. Заказы и контакты

- **`Order`** — пользователь, дата, **статус** (`OrderState`: `basket`, `new`, `confirmed`, …), опционально **`Contact`** после оформления.
- **`OrderItem`** — ссылка на `ProductInfo` и количество; уникальность пары `(order, product_info)`.
- **`Contact`** — до **пяти** адресов на пользователя (ограничение проверяется в API); поля города, улицы, телефона и др.

### 3.4. Регистрация

- **`ConfirmEmailToken`** — одноразовый ключ, выдаётся при создании неактивного пользователя и удаляется после успешного подтверждения.

## 4. Потоки данных

### 4.1. Корзина и подтверждение заказа

1. Для каждого покупателя поддерживается заказ со статусом **`basket`** (создаётся при первом обращении к корзине).
2. Позиции — строки **`OrderItem`** с ссылкой на **`ProductInfo`** (конкретный магазин и цена зафиксированы в позиции).
3. Подтверждение: у заказа-корзины выставляются **`contact_id`** и статус **`new`**, вызывается **`save()`** (не массовый `update`), чтобы отработали **сигналы** и ушли письма клиенту и администратору.

### 4.2. Импорт прайса

- Синхронно: **`POST /api/v1/partner/update`** с телом `url` — загрузка YAML по HTTP и разбор в **`services/yaml_catalog.py`**.
- Локально: команда **`import_shop_yaml`** читает файл с диска.
- Асинхронно (проверка продвинутой части): **`POST /api/v1/admin/import_task`** ставит в очередь Celery задачу **`do_import_task`**.

### 4.3. Экспорт

- **`GET /api/v1/partner/export`** — формирование YAML текущего каталога магазина из БД (**`services/yaml_export.py`**).

## 5. Интеграции

- **Django REST Framework** — рендер JSON, пагинация списков (`PageNumberPagination`, размер страницы из настроек), аутентификация по **Token** (`rest_framework.authtoken`).
- **django-rest-passwordreset** — эндпоинты сброса пароля подключены в **`zakupki/urls.py`** (не внутри `catalog.urls`).
- **Celery + Redis** — брокер и backend результатов задаются в `settings`; включение фоновой отправки писем при смене статуса — флаг **`USE_CELERY_FOR_SLOW_OPS`**.

## 6. Админ-панель

Зарегистрированы основные сущности. Для **`Order`** включено **`list_editable`** по полю **`state`**: сохранение строки в списке вызывает **`save()`** модели и срабатывание сигналов — клиент получает уведомление о смене статуса (если статус реально изменился).

## 7. Зависимости между слоями

Представления импортируют сериализаторы и модели; тяжёлую логику почты и YAML выносят в **services**; **signals** вызывают **mail_notifications** и при необходимости ставят задачи в **tasks**. Таким образом, смена способа доставки почты (синхронно / Celery) локализована в сигналах и настройках.
