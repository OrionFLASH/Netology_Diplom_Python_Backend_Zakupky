# Логирование и защита персональных данных в логах

## 1. Назначение

Логи фиксируют ключевые действия (регистрация, вход, импорт прайса, подтверждение заказа, ошибки) и помогают отладке. При этом в сообщениях **не должны** в открытом виде фигурировать персональные данные: полные телефоны, email, адреса, идентификаторы пользователей — в соответствии с требованиями к обработке ПДн в учебном проекте.

## 2. Конфигурация

Файл **`zakupki/settings.py`** задаёт:

- каталог **`log/`** (создаётся при старте, если отсутствует);
- два файловых обработчика с именами по шаблону:
  - **`INFO_zakupki_ГГГГММДД_ЧЧ.log`**
  - **`DEBUG_zakupki_ГГГГММДД_ЧЧ.log`**
- логгер **`catalog`** с уровнем **DEBUG**, вывод в консоль и в оба файла.

Имя файла привязано к часу запуска процесса (при перезапуске сервера создаётся новый файл с новым часом).

## 3. Формат строки DEBUG

Класс **`zakupki.logging_utils.StructuredDebugFormatter`** формирует строки вида:

```text
2026-04-03 12:00:00,000 - [DEBUG] - текст сообщения [class: ИмяКласса | def: имя_метода]
```

В коде приложения в **`logger.debug(..., extra={"caller_class": "...", "caller_def": "..."})`** передаётся контекст вызова. Если поля не заданы, подставляются прочерки.

## 4. Маскирование (модуль `catalog.privacy`)

| Функция | Поведение |
|---------|-----------|
| **`mask_email`** | Локальная часть сокращается до вида `a***@domain.ru`. |
| **`mask_phone`** | Остаются только последние 2 цифры после вычищения нецифровых символов. |
| **`mask_address_fragment`** | Короткий префикс строки + `***`. |
| **`mask_user_id`** | В лог не пишется сырой id; используется форма с хэшем по модулю (для ориентира без прямой идентификации). |
| **`safe_contact_log_payload`** | Обходит словарь полей контакта и маскирует известные ключи (email, phone, адресные поля). |

Эти функции применяются в представлениях и сервисах там, где в лог попадают пользовательские вводы.

## 5. Что проверить при приёмке

1. Выполнить сценарий с реальным телефоном и email в контакте.  
2. Открыть последний **`INFO_*.log`** и **`DEBUG_*.log`**.  
3. Убедиться, что **полных** email, телефонов и адресов нет.  
4. Убедиться, что по сообщениям всё ещё можно понять этап (например, «добавлен контакт», «заказ подтверждён»).

## 6. Ограничения

- Тела исходящих **email** формируются для получателей и могут содержать полные данные заказа для **администратора** — это не лог-файл, а доверенный канал доставки накладной.  
- При использовании **консольного** `EMAIL_BACKEND` содержимое писем выводится в stdout процесса — в продакшене используйте настоящий SMTP и ограничьте доступ к логам сервера.
