# Описание REST API

Базовый префикс всех маршрутов приложения: **`/api/v1/`**.  
Корневые URL объявлены в `zakupki/urls.py`, маршруты каталога — в `catalog/urls.py`.

## Общие правила

### Аутентификация

- Для защищённых методов заголовок: **`Authorization: Token <ключ>`**, где ключ возвращается **`POST /api/v1/user/login`** после успешного входа.
- Пользователь с **`is_active=False`** не получает токен (не подтверждён email).

### Формат ответов

- Успешные операции, унаследованные от учебного API, часто возвращают JSON вида **`{"Status": true}`** или **`{"Status": false, "Errors": ...}`**.
- Списочные представления DRF (`categories`, `shops`) возвращают **пагинированный** ответ: `count`, `next`, `previous`, `results`.
- Код **403** с телом **`{"Status": false, "Error": "Log in required"}`** — нет или неверный токен.

### Тип содержимого

Рекомендуется **`Content-Type: application/json`** для тел с JSON. Поле **`items`** в корзине — **строка**, внутри которой лежит JSON (совместимость с учебным клиентом).

## 1. Регистрация и профиль

### `POST /api/v1/user/register`

Регистрация покупателя.

**Тело (JSON):**

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `first_name` | да | Имя |
| `last_name` | да | Фамилия |
| `email` | да | Email (логин) |
| `password` | да | Пароль (проверяется валидаторами Django) |
| `company` | да | Компания |
| `position` | да | Должность |

**Успех:** `{"Status": true}` — пользователь создан с `is_active=false`; в обработчике сигнала создаётся токен и отправляется письмо с ключом подтверждения.

**Ошибки:** `{"Status": false, "Errors": ...}` — неверный набор полей, слабый пароль или ошибки уникальности.

### `POST /api/v1/user/register/confirm`

**Тело:** `email`, `token` (из письма).

**Успех:** `{"Status": true}` — `is_active=true`, токен подтверждения удалён.

### `POST /api/v1/user/login`

**Тело:** `email`, `password`.

**Успех:** `{"Status": true, "Token": "<строка>"}`.

**Неуспех:** `{"Status": false, "Errors": "Не удалось авторизовать"}`.

### `GET /api/v1/user/details`

Требуется токен. Ответ — сериализованный профиль с вложенными **`contacts`**.

### `POST /api/v1/user/details`

Частичное обновление профиля; при наличии **`password`** — смена пароля с проверкой сложности.

## 2. Контакты доставки

### `GET /api/v1/user/contact`

Список контактов текущего пользователя.

### `POST /api/v1/user/contact`

Минимум: **`city`**, **`street`**, **`phone`**. Дополнительно: `first_name`, `last_name`, `patronymic`, `contact_email`, `house`, `structure`, `building`, `apartment`.

Не более **5** контактов на пользователя.

### `PUT /api/v1/user/contact`

Обновление: в теле должен быть **`id`** контакта (принадлежащего пользователю).

### `DELETE /api/v1/user/contact`

Тело: **`items`** — строка с id через запятую, например `"3,5"` (как в учебном API).

## 3. Справочники и каталог

### `GET /api/v1/categories`

Все категории (пагинация DRF).

### `GET /api/v1/shops`

Магазины с **`state=true`** (принимают заказы).

### `GET /api/v1/products`

Query-параметры (опционально):

- **`shop_id`** — фильтр по магазину;
- **`category_id`** — фильтр по категории товара.

В выдаче: `id`, `model`, `product` (имя, категория, описание), `shop`, **`supplier`** (имя магазина), `quantity`, `price`, `price_rrc`, **`product_parameters`** (массив пар имя/значение).

Отдельного URL «один товар по id» нет: детализация — по записи **`ProductInfo`** в списке (можно фильтровать до одной записи на клиенте).

## 4. Корзина

Одна корзина на пользователя — заказ со статусом **`basket`**.

### `GET /api/v1/basket`

Массив заказов-корзин (обычно 0 или 1 элемент) с позициями и полем **`total_sum`** (сумма `quantity * price`).

### `POST /api/v1/basket`

**Тело:** поле **`items`** в одном из двух форматов (совместимость с учебным API и с обычными JSON-клиентами):

1. **Строка**, внутри которой записан JSON-массив (как в исходном примере Netology):

```json
{ "items": "[{\"product_info\": 1, \"quantity\": 2}]" }
```

2. **Массив объектов** в теле `application/json` (удобно для `curl` и фронтенда):

```json
{ "items": [{ "product_info": 1, "quantity": 2 }] }
```

Каждый элемент: **`product_info`** (id `ProductInfo`), **`quantity`**. Поле `order` подставляется сервером.

**Успех:** `{"Status": true, "Создано объектов": N}`.

### `PUT /api/v1/basket`

**Тело:** **`items`** — так же **строка** с JSON-массивом или **готовый массив** объектов **`{"id": <id позиции в корзине>, "quantity": <число>}`**.

### `DELETE /api/v1/basket`

**Тело:** **`items`** — строка id позиций через запятую (`"3,5"`) **или** массив чисел (`[3, 5]`).

## 5. Заказы покупателя

### `GET /api/v1/order`

Список заказов пользователя **без** статуса `basket`, с позициями, **`total_sum`**, контактом.

Query: **`id=<номер>`** — сузить до одного заказа (детальный просмотр).

### `POST /api/v1/order`

Подтверждение заказа из корзины.

**Тело:**

- **`id`** — id заказа-корзины (строка или число, проверяется как корзина текущего пользователя);
- **`contact`** — id контакта доставки.

После успеха статус **`new`**, отправляются письма (клиент + администратор при заданном `ADMIN_ORDER_EMAIL`).

### `PUT /api/v1/order`

Только для **`is_staff`**.

**Тело:** **`id`** заказа, **`state`** — одно из: `new`, `confirmed`, `assembled`, `sent`, `delivered`, `canceled` (не `basket`).

При сохранении модели уходит уведомление клиенту о смене статуса (через сигнал).

## 6. API магазина (тип пользователя `shop`)

Все методы требуют токен пользователя с **`type=shop`**.

### `POST /api/v1/partner/update`

**Тело:** **`url`** — ссылка на YAML-прайс. Проверка URL, загрузка, разбор и запись в БД.

### `GET /api/v1/partner/state` / `POST /api/v1/partner/state`

Текущий объект магазина; **POST** — поле **`state`**: строка `"true"` / `"false"` (или `1`/`0`).

### `GET /api/v1/partner/orders`

Заказы (не корзины), в которых есть товары этого магазина.

### `GET /api/v1/partner/export`

Ответ: файл **`application/x-yaml`**, каталог текущего магазина.

## 7. Административное API

### `POST /api/v1/admin/import_task`

Только **`is_staff`** / суперпользователь.

**Тело:** **`url`**, **`shop_user_id`** (числовой id пользователя-магазина).

**Ответ:** `{"Status": true, "task_id": "<celery-id>"}` — задача **`do_import_task`** поставлена в очередь.

---

## 8. Сброс пароля (django-rest-passwordreset)

Маршруты в **`zakupki/urls.py`**:

- **`POST /api/v1/user/password_reset`** — запрос токена (формат полей см. документацию пакета и Postman-коллекцию курса).
- **`POST /api/v1/user/password_reset/confirm`** — подтверждение с токеном.

При создании токена срабатывает сигнал: письмо с ключом уходит на email пользователя (настройки SMTP в `.env`).

---

## 9. Админ-сайт Django

**`/admin/`** — веб-интерфейс: пользователи, заказы, каталог. Не является частью JSON API, но используется для смены статусов заказов и управления данными.
