Это ближе к полям из **`screens.md`** (ФИО, email, телефон, адрес), чем к строгому разделению двух типов сущностей в БД.

## 3. Пункты этапа 4 из корневого README задания

Требование: реализованы views для основных страниц **без админки** (список из README Netology):

| Пункт задания | В проекте |
|---------------|-----------|
| Авторизация | `POST /api/v1/user/login`, токен DRF (`views_auth.py`). |
| Регистрация | `POST /user/register`, `POST /user/register/confirm` (+ расширенные поля `company`, `position` как в учебной схеме регистрации). |
| Список товаров | `GET /categories`, `/shops`, `/products` с фильтрами `shop_id`, `category_id` (`views_catalog.py`). |
| **Спецификация по отдельному товару** | Отдельного URL вида `/products/<id>` **нет**. Каталог отдаёт список записей **`ProductInfo`**; клиент выбирает позицию по **`id`** из ответа. Детализация заказа: **`GET /order?id=…`**. В **`Docs/API.md`** это явно зафиксировано. Для буквального дублирования некоторых Postman-сценариев при необходимости можно добавить отдельный эндпоинт детальной карточки. |
| Корзина (добавление, удаление) | `GET/POST/PUT/DELETE /basket` (`views_basket.py`), в т.ч. `items` строкой или массивом. |
| Адрес доставки | `GET/POST/PUT/DELETE /user/contact`, лимит 5 записей (`views_contact.py`). |
| Подтверждение заказа | `POST /order` (корзина → статус `new`). |
| Email с подтверждением | Письма при регистрации, при подтверждении заказа, накладная администратору (`mail_notifications.py`, `signals.py`, опционально Celery). |
| Список заказов | `GET /order`. |
| Детали заказа | `GET /order?id=<номер>`. |
| Редактирование статуса заказа | `PUT /order` для **`is_staff`** (`views_order.py`). В README этап 4 относит это к базовой части; у вас это же API, плюс Django Admin для ручного изменения. |
| Восстановление пароля (из общего описания пользователей в README задания) | `POST /api/v1/user/password_reset` и `/confirm` (`zakupki/urls.py`, пакет `django-rest-passwordreset`). |

## 4. Продвинутая часть (по README Netology)

| Требование | Реализация |
|------------|------------|
| **Экспорт товаров** | `GET /api/v1/partner/export`, `yaml_export.py`. |
| **Админка заказов, статусы, уведомление клиента** | `src/catalog/admin.py` (заказы и позиции); смена статуса в админке и через **`PUT /order`**; сигнал отправки письма при смене статуса (`signals.py`). |
| **Медленные операции асинхронно (email, импорт, экспорт)** | Celery: `zakupki/celery.py`, задачи **`send_email_task`**, **`do_import_task`**, а также отложенные `send_order_notifications_task`, `send_order_status_changed_task` (`tasks.py`). Включение через настройки (`USE_CELERY_FOR_SLOW_OPS` и др. в `settings.py`). |
| **View для запуска `do_import` «из админки»** | В задании формулировка смешивает админку Django и «админский» сценарий. Здесь: **`POST /api/v1/admin/import_task`** (`AdminImportTaskView`, только staff) вызывает **`do_import_task.delay`**. Отдельной кнопки в шаблоне Django Admin под эту задачу нет — запуск через API персонала. |
| **Docker Compose + инструкция** | `docker-compose.yml`, `Dockerfile`, разделы в **`Docs/DEPLOYMENT.md`**. |

**Замечание по этапу 6 в README Netology:** в корневом README указано «API views админки склада» и ссылка на `step-6-adv.md`, а в **`step-6-adv.md`** описана именно **Django Admin**, а не отдельный REST для «склада». В данном проекте есть и **Django Admin**, и **партнёрское API** для магазина — это покрывает практический смысл этапа.

## 6. Итоговая оценка

- **Исходные данные** из списка README задания учтены; формат YAML и идея API согласованы с [репозиторием курса](https://github.com/netology-code/python-final-diplom/tree/master).
- **Базовая часть** по смыслу реализована; единственная явная зона уточнения у проверяющего — **отсутствие отдельного эндпоинта «одна позиция каталога по id»**, при этом данные позиции доступны через **`GET /products`** и **`id` в корзине/заказе**.
- **Продвинутая часть** в объёме README Netology **реализована** (экспорт, админка + уведомления, Celery, docker-compose), с нюансом: **`do_import`** запускается через **REST для staff**, а не через встроенную форму Django Admin.

При приёмке работы уточните у дипломного руководителя, достаточно ли текущей схемы детализации товара и способа вызова Celery-импорта.
