# Тестирование

## 1. Автотесты (pytest-django)

### 1.1. Установка и запуск

Из корня репозитория с активированным виртуальным окружением:

```bash
pip install -r requirements.txt
pytest src/Tests -v
```

Краткий режим:

```bash
pytest src/Tests -q
```

Конфигурация: **`pytest.ini`** (`DJANGO_SETTINGS_MODULE=zakupki.settings`, `pythonpath = . src`).

### 1.2. Фикстуры

**`src/Tests/conftest.py`** подменяет **`EMAIL_BACKEND`** на **`locmem`**, чтобы в тестах можно было проверять **`django.core.mail.outbox`**.

### 1.3. Сценарий `test_buyer_registration_login_and_order`

Файл **`src/Tests/test_buyer_flow.py`**. Последовательность:

1. Регистрация покупателя через **`POST /api/v1/user/register`**.  
2. Подтверждение email токеном из модели **`ConfirmEmailToken`**.  
3. Вход, получение **`Token`**.  
4. Создание пользователя **`shop`**, импорт **`data/shop1.yaml`** через **`import_price_from_bytes`**.  
5. Добавление позиции в корзину (**`POST /basket`**, поле **`items`** — JSON-строка).  
6. Создание контакта.  
7. Подтверждение заказа (**`POST /order`**).  
8. Проверка, что в **`mail.outbox`** есть хотя бы одно письмо.  
9. Проверка списка заказов (**`GET /order`**) на наличие статуса **`new`**.

### 1.4. Запуск тестов в Docker

```bash
docker compose run --rm web pytest src/Tests -v
```

Требуется собранный образ **`web`** и доступность Redis, если compose поднимает зависимости автоматически для `run`.

## 2. Ручная проверка API

Рекомендуется **Postman** или **curl**; ориентир по наборам запросов — документация Postman из репозитория курса (ссылка в **`Docs/README_istochniki.md`**).

Минимальный чек-лист:

| Шаг | Действие | Ожидание |
|-----|----------|----------|
| 1 | Импорт прайса для магазина | В БД появляются категории и `ProductInfo`. |
| 2 | `GET /api/v1/products` | JSON со списком предложений. |
| 3 | Регистрация покупателя | `Status: true`, пользователь неактивен. |
| 4 | Подтверждение email | Пользователь активен, вход выдаёт токен. |
| 5 | Корзина POST/GET | Позиции и `total_sum`. |
| 6 | Контакт POST | `Status: true`, не более 5 контактов. |
| 7 | Заказ POST | `Status: true`, письма (SMTP или консоль). |
| 8 | Заказ GET | Список без корзины, статус `new`. |
| 9 | Админка: смена статуса заказа | Письмо клиенту о смене статуса. |
| 10 | Партнёр: export GET | Файл YAML. |

## 3. Проверка логов

После сценария с контактом откройте **`log/DEBUG_*.log`** и **`log/INFO_*.log`**: убедитесь в отсутствии полных email/телефонов в строках, где используется маскирование (см. **`Docs/LOGGING.md`**).

## 4. Проверка Celery

1. Запустить Redis и воркер: **`celery -A zakupki worker -l info`**.  
2. В `.env`: **`USE_CELERY_FOR_SLOW_OPS=True`**.  
3. Подтвердить заказ или вызвать **`POST /admin/import_task`** — задачи должны появиться в логе воркера без ошибок импорта **`catalog`**.
